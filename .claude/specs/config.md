# Crossler — Config Format

Рабочий документ по проектированию формата конфигурационного файла и CLI.
Содержимое — текущее понимание и ориентиры, не финальная спецификация.

---

## Группы данных конфига

Предварительный список логических групп. Порядок и состав будут уточняться.

### 1. `metadata`
Общая информация о продукте: название, версия, описание, лицензия, мейнтейнер, сайт и т.д.

### 2. `binaries`
Пути к бинарникам по таргетам (ОС + архитектура), куда их устанавливать.
Выделены отдельно, потому что:
- подписываются *до* упаковки
- имеют платформо-зависимое расширение (`.exe` на Windows)
- путь к артефакту сборки зависит от таргета

### 3. `files`
Все остальные файлы, симлинки, директории.
Каждый элемент описывает: источник, путь назначения, тип (`file`, `symlink`, `dir`), атрибуты (`config`, `readonly`, `noreplace` и т.д.).
Симлинки — тип файла, не отдельная группа.
Флаги — атрибут конкретного файла, не отдельная группа.

### 4. `scripts`
Хуки жизненного цикла пакета: preinstall, postinstall, preremove, postremove.

> **TODO:** Требует отдельного детального обдумывания — какие именно хуки нужны, в каком формате задаются (путь к скрипту или inline-команда), как работают на разных платформах и форматах пакетов.

### 5. `dependencies`
Зависимости от системных пакетов. В первую очередь актуально для Linux (`.deb`, `.rpm`).

### 6. `signing`
Параметры подписания артефактов:
- Windows Authenticode (через `osslsigncode` или `signtool`)
- Apple Code Signing + нотаризация (через `codesign`/`notarytool` или `rcodesign`)

### 7. `integration`
Системная интеграция. Внутри — подсекции:
- сервисы / демоны (systemd, launchd, Windows Service)
- переменные окружения
- ярлыки (Windows shortcuts)
- контекстное меню

### 8. `platforms`
Переопределения для конкретных ОС (`linux`, `macos`, `windows`) или форматов пакетов (`deb`, `rpm`, `msi` и т.д.).
Базовые настройки работают для всех платформ; здесь — только то, что нужно изменить.

## Сравнительный анализ: идеи Crossler vs. референсные инструменты

### Источники для сравнения

| Инструмент | Что изучалось |
|------------|---------------|
| **GoReleaser** | `.goreleaser.yaml` — единый конфиг релиза; структура секций builds/archives/nfpms/brews/signs; подход к шаблонам и переопределениям |
| **nfpm** | `nfpm.yaml` — конфиг Linux-упаковщика; модель contents/scripts/overrides; типы файлов (config, symlink, dir, ghost, tree) |
| **wixl** | `.wxs` XML — структура MSI; UpgradeCode, компоненты, CustomAction, реестр, ярлыки |
| **pkgbuild / hdiutil** | Подход macOS; component plist; DMG-форматы |

---

### 1. Плоская vs. секционная структура конфига

**GoReleaser** использует радикально секционный подход: каждый тип артефакта — отдельная топ-уровневая секция (`builds`, `archives`, `nfpms`, `pkgs`, `dmgs`, `msi`, `brews`, `signs`, …). Пользователь описывает процесс как конвейер — шаг за шагом.

**Crossler** (текущий черновик) использует плоский подход: метаданные, файлы, скрипты, зависимости на верхнем уровне, платформенная специфика — в `platforms.*`. Упор не на процесс (что делать), а на описание продукта (что установить).

**Оценка:** подход Crossler правильный для его задачи. GoReleaser — это инструмент релиза (сборка + упаковка + публикация + анонс); Crossler — только упаковщик. Два конфига с разным охватом. Плоская структура Crossler проще и удобнее для целевого сценария.

**Риск:** в GoReleaser каждая секция имеет `id:` и позволяет иметь несколько вариантов (например, два разных `nfpms` с разными фильтрами). В Crossler сейчас предполагается один конфиг — один продукт. Это правильное ограничение, но нужно явно зафиксировать.

---

### 2. Переопределения (override) — уровни и механика

**nfpm** поддерживает `overrides:` на уровне формата (deb/rpm/apk/archlinux). Переопределить можно практически любое поле.

**GoReleaser/nfpm** поддерживает отдельные `contents:` внутри `overrides:` — т.е. можно добавить файлы только для deb, но не для rpm.

**Crossler** (текущий черновик) задумывает 3 уровня: глобальный → платформа → формат. Это более гибко чем nfpm и покрывает реальные нужды. Предложенный `[[platforms.linux.files]]` — отдельный массив файлов именно для Linux.

**Новая идея из анализа:** GoReleaser в `archives.format_overrides` переопределяет формат архива по ОС:
```yaml
format_overrides:
  - goos: windows
    formats: [zip]
```
Аналогичное для Crossler — переопределение набора форматов пакетов по платформе — уже есть неявно (msi только для windows, dmg только для macos), но стоит сделать это явным.

**Итог:** текущий подход Crossler (три уровня + платформенные массивы файлов) охватывает нужды nfpm и GoReleaser. Дополнительно нужно подумать об уровне архитектуры как отдельном переопределении.

---

### 3. Описание файлов: `files` vs. `contents`

**nfpm** называет секцию `contents:` (что правильно отражает «содержимое пакета»). GoReleaser в `archives` и `nfpms` использует одинаковую модель со структурой `{src, dst, type, file_info}`.

**Crossler** (текущий черновик) использует `[[files]]` с полями `src`, `dst`, `win_dst`, `type`, `mode`, `owner`, `group`.

**Сравнение типов файлов:**

| Тип | nfpm | Crossler (черновик) | Нужен? |
|-----|------|---------------------|--------|
| `file` | ✓ | ✓ | Да |
| `config` | ✓ | ✓ | Да |
| `config\|noreplace` | ✓ | ✓ | Да |
| `symlink` | ✓ | ✓ | Да |
| `dir` | ✓ | ✓ | Да |
| `doc` | ✓ | ✓ | Да |
| `license` | ✓ | ✓ | Да |
| `ghost` | ✓ (RPM) | ✓ (в черновике) | Спорно — только RPM, нишевой |
| `tree` | ✓ | ✓ (в черновике) | Да — удобно для рекурсивных директорий |

**Наблюдение:** `win_dst` как отдельное поле — специфика Crossler, которой нет ни в nfpm, ни в GoReleaser. Оба инструмента описывают содержимое для одной целевой платформы. Crossler уникален тем, что один конфиг покрывает все платформы, включая Windows, у которой другая структура путей.

**Вывод:** набор типов в черновике полный. `ghost` — на усмотрение (только RPM, редко нужен). `tree` — полезен. Вопрос `win_dst` требует отдельного решения (см. раздел «Открытые вопросы»).

### 4. Шаблоны и переменные

**GoReleaser** использует Go-шаблоны (`{{ .Version }}`, `{{ .Os }}`, `{{ .Arch }}`) — мощно, но требует знания синтаксиса шаблонов Go.

**nfpm** поддерживает простые переменные в путях (через `expand: true`).

**Crossler** (текущий черновик) предлагает простую подстановку `${VARS}`: `${VERSION}`, `${GOOS}`, `${GOARCH}`, `${EXE}`, `${BUILD_NUMBER}`.

**Оценка:** простой `${VAR}` синтаксис — правильный выбор для Crossler. Он понятнее Go-шаблонов для целевой аудитории. GoReleaser перегружен шаблонами, что является одной из жалоб пользователей. Для Crossler достаточно подстановки переменных без условий и функций.

**Потенциальная проблема:** как будет выглядеть `src` для Windows-бинарника с `.exe`? `dist/${GOOS}_${GOARCH}/myapp${EXE}` — решение из черновика выглядит чисто.

---

### 5. Скрипты жизненного цикла

**nfpm** поддерживает: `preinstall`, `postinstall`, `preremove`, `postremove` (Linux). Для RPM дополнительно: `pretrans`, `posttrans`. Для apk/archlinux: `preupgrade`, `postupgrade`.

**GoReleaser** (nfpms.scripts) — та же модель nfpm плюс переопределения по формату.

**pkgbuild** поддерживает только `preinstall` и `postinstall`.

**wixl** поддерживает `CustomAction` (только EXE-based, не shell-скрипты).

**Crossler** (текущий черновик) предлагает 4 хука: `preinstall`, `postinstall`, `preremove`, `postremove`.

**Новые хуки из анализа:**
- `pretrans` / `posttrans` (RPM) — нишевые, не нужны в v1
- `preupgrade` / `postupgrade` (apk, archlinux) — нишевые, не нужны в v1

**Оценка:** 4 хука в черновике — правильный минимум. Расширенные RPM-хуки можно добавить в `platforms.linux.formats.rpm.scripts` при необходимости.

---

### 6. Зависимости пакетов

**nfpm** поддерживает: `depends`, `provides`, `recommends`, `suggests`, `conflicts`, `replaces`. Синтаксис для разных форматов разный (deb: `bash (>= 4.0)`, rpm: `bash >= 4.0`, apk: `bash>=4.0`). nfpm нормализует базовую форму автоматически.

**Crossler** (текущий черновик в `toml-example.toml`) предлагает ту же модель: `depends`, `recommends`, `suggests`, `conflicts`, `replaces`, `provides`. Это правильно.

**Вывод:** полный набор полей зависимостей из nfpm необходим. Crossler должен нормализовать синтаксис: принимать нейтральную форму `bash >= 4.0` и генерировать правильный синтаксис для каждого формата. Переопределения с форматным синтаксисом через `platforms.linux.formats.deb.depends` — как escape-люк.

---

### 7. Windows: реестр, ярлыки, ассоциации файлов

**wixl** поддерживает регистрацию в реестре (`<RegistryKey>`), ярлыки (`<Shortcut>`), ассоциации файлов (`<ProgId>`, `<Extension>`). Это нативные возможности WiX.

**GoReleaser Pro (msi)** — делегирует WiX, поддержка этих возможностей ограничена.

**Crossler** (текущий черновик): реестр, ярлыки, ассоциации файлов описаны в `platforms.windows.formats.msi.*`.

**Оценка:** решение правильное. Эти возможности должны быть в секции `platforms.windows.formats.msi`, а не на верхнем уровне — они строго Windows/MSI-специфичны. GoReleaser явно недостаточен в этом месте (Pro-версия, ограниченная поддержка); Crossler здесь превосходит.

---

### 8. macOS: `.pkg` и `.dmg`

**pkgbuild** требует: identifier (обратная DNS-нотация), version, скрипты (preinstall/postinstall). Опционально: component plist для .app-бандлов.

**hdiutil**: создаёт DMG из staging-директории. Кастомизация (фон, позиции иконок) требует AppleScript или сторонних инструментов — недоступна в headless CI.

**GoReleaser Pro** поддерживает оба формата нативно.

**Crossler** (текущий черновик) для `.pkg` предлагает `identifier` на уровне `platforms.macos`. Для `.dmg` — `format`, `volume_name`, `background_image`, `icon_size`, `window_size`, `icon_positions`.

**Критическое замечание по DMG:** кастомизация (background_image, icon_positions) требует запущенного Finder и AppleScript. Это недоступно в headless Linux/Windows CI, где чаще всего и собирается. Это либо macOS-only функционал, либо реализуется через сторонние инструменты (`dmgbuild`). Нужно чётко обозначить ограничение.

**Оценка:** `identifier` для pkg — обязателен. Параметры DMG-кастомизации — чрезмерно сложны для v1 (противоречие с CLAUDE.md, где DMG-кастомизация в категории «не нужно»). Стоит ограничиться `format` и `volume_name`.

---

### 9. Homebrew: formula vs. cask

**GoReleaser** различает `brews` (CLI-инструменты, formula) и `homebrew_casks` (GUI-приложения с .app, cask). Для formula: автоматически генерирует блок `install`, `test`, `service` (для демонов). Для cask: отдельная схема с `pkg`, `dmg`, `app`.

**Crossler** (текущий черновик) поддерживает только formula (`[homebrew]`). Cask не нужен (GUI-приложения — вторичная аудитория).

**Детали из GoReleaser, которые стоит взять:**
- `test_command` / `test:` — команда проверки после установки Homebrew (у GoReleaser `test:` блок в Ruby, у Crossler `test_command` проще)
- `service:` в Homebrew formula — для демонов, которые должны запускаться через launchd. Полезно, но не срочно для v1.
- `skip_upload: auto` — пропускать для pre-release тегов. Концепция стоит адаптации.
- `install_extras` (Crossler) соответствует `extra_install:` в GoReleaser — оба решают одну задачу.

**Оценка:** текущий подход Crossler (`[homebrew]`) покрывает 95% сценариев. Service-поддержка и `skip_upload` — для v1.x.

---

### 10. Подпись артефактов

**GoReleaser** использует универсальную модель `signs:` с командой и аргументами — работает с GPG, cosign и любым другим инструментом. Отдельно: `binary_signs:` для бинарников, `notarize:` для macOS.

**Crossler** (текущий черновик) использует специализированные секции `[sign_windows]` и `[sign_macos]` с конкретными полями для каждой системы подписи.

**Сравнение:**

| Аспект | GoReleaser | Crossler |
|--------|-----------|---------|
| Модель | Универсальная: команда + аргументы | Специализированная: конкретные поля |
| Windows Authenticode | Через кастомную команду | `[sign_windows]` с `certificate`, `password`, `timestamp_url` |
| Apple Code Signing | `notarize.macos` | `[sign_macos]` с `certificate`, `identity`, `entitlements`, `notarize` |
| GPG / cosign | ✓ (основной сценарий) | — (не нужен для целевой аудитории) |
| Выбор инструмента | Явный (`cmd: cosign`) | Автоматический по хост-платформе |

**Оценка:** специализированный подход Crossler лучше для пользователя — не нужно знать аргументы командной строки osslsigncode или rcodesign. GoReleaser универсален, но требует от пользователя больше знаний. Автоматический выбор инструмента по хост-платформе (osslsigncode на Linux, signtool на Windows) — ключевое преимущество Crossler.

**Что взять из GoReleaser:**
- Концепция `if:` (условное выполнение) — полезна для `sign_windows` / `sign_macos` (пропускать если сертификат не задан). В Crossler это можно реализовать проще: секция просто игнорируется если `certificate` не указан.

### 11. Выходные файлы и именование артефактов

**GoReleaser** предоставляет шаблоны имён (`name_template`) для всех типов артефактов: архивы, пакеты, checksums. Поддерживает сложные условия через Go-шаблоны.

**Crossler** (текущий черновик) не затрагивает именование выходных файлов.

**Открытый вопрос для Crossler:** как будут называться итоговые файлы?
- `myapp_1.0.0_linux_amd64.tar.gz` — GoReleaser-стиль
- `myapp-1.0.0-1.x86_64.rpm` — nfpm-стиль (следует конвенции формата)

**Рекомендация:** для Linux-пакетов (deb/rpm/apk) использовать нативные конвенции именования каждого формата — nfpm делает это автоматически. Для tar.gz, msi, pkg, dmg — единый шаблон вида `{name}_{version}_{os}_{arch}.{ext}`.

---

### 12. Контрольные суммы (checksums)

**GoReleaser** автоматически генерирует файл checksums (`sha256`) для всех артефактов. Настраивается через `checksum:`.

**Crossler** (текущий черновик) не описывает checksums.

**Рекомендация:** автоматически генерировать `sha256sums.txt` (или `{name}_{version}_checksums.txt`) после сборки всех пакетов. Это нужно для верификации скачанных файлов и для Homebrew formula (требует SHA256 для каждого архива). Добавить как неявное поведение, управляемое флагом `checksums = true` на верхнем уровне.

---

### 13. Порядок шагов (pipeline)

**GoReleaser** явно описывает pipeline из ~30 шагов — пользователь видит процесс целиком.

**Crossler** скрывает pipeline от пользователя — это правильно. Но внутренний порядок должен быть детерминированным:

1. Валидация конфига
2. Подготовка staging-директории
3. Подпись бинарников (до упаковки)
4. Сборка пакетов (параллельно по форматам)
5. Подпись пакетов (после сборки, до публикации)
6. Нотаризация (только macOS)
7. Staple (только macOS, после нотаризации)
8. Генерация checksums
9. Генерация Homebrew formula

Этот порядок важен — особенно п.3 (подпись бинарника до упаковки) и п.5-7 (подпись/нотаризация пакета). Стоит зафиксировать в спецификации.

### Итог: что взять в Crossler из анализа

#### Взять без изменений
- Набор типов файлов nfpm: `file`, `config`, `config|noreplace`, `symlink`, `dir`, `doc`, `license`, `tree`
- Модель зависимостей nfpm: `depends`, `recommends`, `suggests`, `conflicts`, `replaces`, `provides`
- Homebrew `test_command` и `install_extras` (аналог GoReleaser `test:` / `extra_install:`)
- Автоматическое именование Linux-пакетов по конвенциям каждого формата

#### Взять с адаптацией
- Три уровня переопределений (глобальный → платформа → формат) — уже есть в черновике, подтверждается нfpm/GoReleaser
- Автоматическая генерация checksums (GoReleaser делает по умолчанию — Crossler тоже должен)
- `skip_upload`-аналог для Homebrew formula (пропускать для pre-release)
- Явный пайплайн шагов с документированным порядком (внутренняя архитектура)

#### Не брать (несмотря на наличие в GoReleaser)
- Go-шаблоны (`{{ .Version }}`) — слишком сложно, `${VAR}` достаточно
- Универсальная модель `signs:` с произвольными командами — Crossler специализирует подпись
- DMG-кастомизация (фон, позиции иконок) — headless CI несовместимо
- `ghost`-тип файлов (RPM-специфика, нишевой) — отложить
- Changelog, announce, blobs, Docker, Ko — вне скоупа Crossler

#### Открытые вопросы, требующие решения

1. **Именование артефактов:** нужен ли шаблон `name_template` для tar.gz / msi / pkg / dmg?
2. **Checksums:** генерировать автоматически или через явный флаг в конфиге?
3. **`win_dst` vs. автомаппинг:** держать отдельное поле или автоматически маппить Unix-пути на Windows?
4. **`ghost` тип файлов:** включить для RPM или отложить?
5. **DMG-кастомизация:** ограничиться `format` + `volume_name`, или поддержать background_image/icon_positions только для macOS-хоста?
6. **Homebrew `service:`:** нужен ли для CLI-утилит, которые могут запускаться как демоны?
